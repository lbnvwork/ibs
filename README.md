# ТЗ

## Общая цель проекта

---
  Проект направлен на создание системы для мониторинга состояния пациентов, принимающих различные препараты, 
  с возможностью интеграции данных из разных источников, таких как кабинет пациента, ЕГИЗ, медицинские гаджеты и кабинет врача.

**Риски и ограничения**
-

- Внесение в ранние итерации большого количества функционала
    <details><summary>Решения:</summary> 
    
    - В первой итерации добиться релиза (публикации на боевом сервере) **минимального** функционала, с которым можно работать пользователям
    - Во второй итерации реализовать **обязательного** функционала для полноценной работы приложения
    - В третьей итерации реализовать весь требуемый функционал
    - Итерации также необходимо разделить на блоки, реализация которых приведет к получению законченной функциональности, несущей ценность для пользователя.
    - Необходимо проанализировать техническое задание вплоть до этих блоков, и выделить основные задачи в рамках блоков (которые могут быть дополнены)
    </details>
- Проблемы с разрешением на получение персональных данных из внешних источников или внесение в них
  <details><summary>Решения:</summary>

  - Выявить проблемы как можно раньше, чтобы решить административные вопросы до начала реализации
  - Изучить документацию внешних источников, опыт подобного внедрения в других проектах
  </details>
- Изменение требований к функционалу в процессе разработки
    <details><summary>Решения:</summary>

    - Необходимо перейти к практике утверждения итерации и блока: 
      - после утверждения итерации, в нее нельзя добавлять новые блоки;
      - после утверждения блока в него нельзя добавлять новые задачи.
    </details>
- Инциденты, связанные с ошибками в реализованном функционале при добавлении нового
    <details><summary>Решения:</summary>
    
    - Проводить внутренней ручное тестирование нового функционала по завершению каждой задачи, 
    - Проводить внутреннее и внешнее (пользовтельское) тестирование всего функционала по завершению каждого блока, каждой итерации
    - Покрытие функционала автоматическими тестами трех уровней: юнит, функциональные, интеграционные
    </details>
- Отсутствие верстальщика
    <details><summary>Решения:</summary>
    
    - Выполнить верстку разработчиком (увеличение времени разработки)
    - Использовать генераторы (сложности с нестандартной версткой, графиками)
    - Найти верстальщика (финансовые риски)
    </details>
- Отсутствие аналитика
    <details><summary>Решения:</summary>

    - Поддержка аналитики разработчиком (увеличение времени разработки, описание задачи не соответствует бизнес-логике приложения)
    - Поддержка аналитики заказчиком (описание задачи и сроки реализации не соответствуют техническим требованиям приложения)
    - Найти аналитика (финансовые риски, время на внедрение в бизнес-логику проекта)
    </details>
- Отсутствие тестировщика
  <details><summary>Решения:</summary>
    
    - Поддержка тестирования разработчиком (увеличение времени разработки, пропуск отрицательных тестовых кейсов - эффект замыленного глаза)
    - Написание автотестов (увеличение времени разработки, сложная, нестандартная логика может не поддаваться автоматическому тестированию)
    - Найти тестировщика (финансовые риски)
  </details>
- Недостаточное количество времени на разработку
  <details><summary>Решения:</summary>
  
    - Увеличение времени разработки (финансовые риски)
    - Освобождение разработчика от обязанностей аналитики, тестирования (финансовые риски, организационны риски)
    - Увеличение количества разработчиков (финансовые риски, организационные риски)
    - Дедлайны на задачи, блоки, итерации (риски, связанные с выгоранием, конфликтами)
    - Четкая постановка задач, без изменений и дополнений (увеличение нагрузки на аналитику, принятие практики запрета на внесение изменений в утвержденные задачи, блоки, итерации)
  </details>
